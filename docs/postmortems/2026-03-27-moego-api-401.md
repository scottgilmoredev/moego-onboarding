# Postmortem — MoeGo API 401: Double-Encoded API Key

**Date:** 2026-03-27
**Severity:** P1 — Critical
**Status:** Complete
**Author:** scottgilmoredev

---

## Summary

All three MoeGo API calls failed with 401 in production. The `MOEGO_API_KEY` script property was already Base64-encoded when stored. `Utilities.base64Encode(apiKey)` in `buildAuthHeader` double-encoded it, producing an invalid credential on every request.

---

## Timeline

| Time       | Event                                                                                 |
| ---------- | ------------------------------------------------------------------------------------- |
| 2026-03-26 | `btoa` unavailable in GAS V8 runtime discovered on first live webhook delivery        |
| 2026-03-26 | `btoa` replaced with `Utilities.base64Encode` throughout codebase                     |
| 2026-03-27 | First full flow execution reached MoeGo API calls — all three returned 401            |
| 2026-03-27 | GCP logs confirmed auth failure on all three calls                                    |
| 2026-03-27 | Root cause identified — API key in Script Properties was already Base64-encoded       |
| 2026-03-27 | Fixed by removing re-encoding — auth header constructed as `Basic ${apiKey}` directly |
| 2026-03-27 | MoeGo API calls successful on next test run                                           |

---

## Root Cause

The `MOEGO_API_KEY` value stored in GAS Script Properties was already Base64-encoded. `buildAuthHeader` applied `Utilities.base64Encode` to it again, producing a doubly-encoded credential that MoeGo's auth layer rejected.

---

## Contributing Factors

- The switch from `btoa` to `Utilities.base64Encode` was made to resolve a separate runtime issue and introduced this regression
- No documentation existed specifying whether the API key should be stored raw or pre-encoded
- GCP logging was not yet set up when `btoa` was first replaced, delaying detection of the 401 until the next full flow execution

---

## Impact

All MoeGo API calls failed for the duration of the affected deployments. Every triggered webhook resulted in a full failure email to the business owner rather than a successful onboarding link delivery.

## Lessons Learned

When replacing a credential encoding function, verify the expected input format of the credential itself — not just the functional equivalence of the replacement. Setup documentation for any credential stored in Script Properties must specify whether the value should be raw or pre-encoded.
